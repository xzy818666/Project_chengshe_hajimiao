## 一、程序功能介绍

### 1.1 项目概述

《赛博功德银行》是一款基于 **Qt 6** 框架开发的桌面金融模拟经营游戏。游戏以"古风仙侠"为视觉包装，以"现代金融"为核心玩法，玩家通过点击法器积累基础货币"功德"，并在多资产市场中进行投资、借贷与风控管理。游戏特色在于将传统点击放置玩法与金融模拟深度结合，加入了**修炼等级、法器共鸣、信用等级动态评级、业障因果系统**等原创机制，形成了一套完整的"点击—积累—投资—风控—成长"闭环。

### 1.2 核心玩法循环

```
功德堂点击法器 → 善财司存贷/借贷 → 幻缘所资产配置 → 市场事件波动
     ↑                                              ↓
  修炼升级 ← 法器共鸣 ← 轮回结算 ← 爆仓/风控/调仓 ←┘
```

玩家在主界面"功德堂"中点击法器获取功德，同时积累修炼经验。获得功德后可进入"善财司"进行存贷管理，或进入"幻缘所"进行资产配置。市场事件系统每 100 ms 驱动资产价格波动，玩家需在波动中调整持仓。高杠杆或高风险操作可能导致爆仓，触发轮回结算。完成成就可解锁长期目标。

### 1.3 主要功能模块

| 模块 | 类名 | 核心职责 |
|------|------|---------|
| 轮回入口 | `StartMenu` | 三种难度选择（简单/普通/困难） |
| 功德堂 | `MeritHall` | 法器点击、HUD 显示、建筑导航、全局事件驱动 |
| 善财司 | `BankDialog` | 活期/定期存款、借贷、信用等级、业障忏悔、杠杆管理 |
| 幻缘所 | `ExchangeDialog` | 四类资产买卖、K 线图、CAL 风险-收益图、智能配置、套利扫描、定投 |
| 法器阁 | `ShopDialog` | 11 种法器购买与装备、7 种共鸣组合检测 |
| 成就榜 | `AchievementDialog` | 8 项成就系统，记录关键里程碑 |
| 新手教程 | `TutorialManager` + `TutorialOverlay` | 38 步内嵌遮罩式实时引导 |

### 1.4 特色系统

| 系统 | 核心机制 |
|------|---------|
| **修炼等级** | 10 级称号体系（凡人→无上佛），每次点击积累修为，升级解锁效率/暴击/杠杆/自动收益加成 |
| **法器共鸣** | 位掩码编码 + 位运算 AND 检测，7 种共鸣组合实时计算效果叠加 |
| **信用等级** | S-F 六级动态评级，行为（还款/逾期/爆仓）实时影响利率、额度与手续费 |
| **业障因果** | 业障影响市场波动率，≥1000 触发"天劫"事件，所有资产价格腰斩 |
| **智能配置** | 基于夏普比率的 CAL 风险-收益图，因果私缘理论价偏离扫描 |
| **内嵌教程** | 38 步遮罩式实时引导，角色"玄机子"边玩边学 |

## 二、项目各模块与类设计细节

### 2.1 整体架构

项目采用经典 **MVC 分层架构**：Model 层负责经济内核与状态管理，View 层负责 Qt 界面展示，Controller 层（主循环）负责定时更新与事件分发。所有模块通过 **Qt 信号槽** 机制进行松耦合通信。

```
┌─────────────────────────────────────────┐
│  界面层 (View)                            │
│  MeritHall / BankDialog / ExchangeDialog  │
│  ShopDialog / AchievementDialog / Tutorial│
└─────────────────┬─────────────────────────┘
                  │ 信号槽通信
┌─────────────────┴─────────────────────────┐
│  控制层 (Controller)                       │
│  meritbank.cpp 主循环: 100ms 更新 + 1s 刷新 │
└─────────────────┬─────────────────────────┘
                  │
┌─────────────────┴─────────────────────────┐
│  模型层 (Model)                            │
│  Wallet / Asset / CultivationLevel          │
│  CreditRating / InstrumentResonance         │
│  MarketEvent / PortfolioAdvisor             │
└─────────────────────────────────────────┘
```

### 2.2 启动与主控层

**`meritbank.cpp`** 为程序入口，负责对象装配（`Wallet`、`Assets`、`MarketEvent`、`AchievementManager` 等）、主循环定时器启动（100 ms 资产更新 + 1 s 界面更新）以及教程系统初始化。

**`StartMenu`** 提供难度选择界面，三种难度影响初始功德、通胀率、信用分与最大杠杆：

| 难度 | 初始功德 | 通胀率 | 信用分 | 最大杠杆 |
|------|---------|--------|--------|---------|
| 简单 | 2000 | 0.5% | 800 | 3× |
| 普通 | 1000 | 1.0% | 500 | 2× |
| 困难 | 500 | 2.0% | 200 | 1.5× |

**`MeritHall`** 为主界面，包含：
- 云海点击区（press/release 切换法器贴图）
- 四角建筑热区（**门位置精确热区**，基于 `door_glow` 中心坐标计算）
- 顶部 HUD（功德、资产、负债、净资产、业障、修炼等级增益明细）
- `resizeEvent` 强制保持 16:9 比例并重新计算所有控件位置

### 2.3 经济内核层

**`Wallet`** 是核心单例类，管理功德余额、债务、信用分、业障、下世功德池、修炼等级与持仓。提供 `earn()`/`spend()` 接口，并通过信号发射 `meritChanged`/`debtChanged`/`yezhangChanged` 等通知 UI 更新。

**`Asset`** 为抽象基类，定义了 `update(dt)`、`price()`、`expectedReturn()` 等接口。四个派生类分别实现不同定价机制：

| 资产 | 类名 | 定价机制 | 风险 |
|------|------|---------|------|
| 众生法缘 | `MeritIndex` | 几何布朗运动，drift=0.02，vol=0.15 | 中 |
| 善行福缘 | `KarmaBond` | 价格恒定为 1.0，持有按秒产生固定收益（年化 4%） | 低 |
| 因果私缘 | `DharmaFund` | 0.6×MeritIndex + 0.4×KarmaBond + 噪声 | 中低 |
| 轮回孽缘 | `SamsaraFutures` | 独立随机游走，vol×3，自带 2× 杠杆，维持率<30% 强制平仓 | 高 |

### 2.4 特色机制层

**`CultivationLevel`** 管理修炼等级。经验曲线呈指数增长：Lv1 需 100，Lv2 需 150，Lv3 需 225… 每次点击获得 1 点经验，暴击额外 5 点。升级奖励：

| 增益类型 | 每级加成 | 上限 |
|---------|---------|------|
| 效率系数 | +2% | 200% |
| 暴击率 | +1% | — |
| 最大杠杆 | +0.1× | — |
| 自动收益 | +3% | — |

**`InstrumentResonance`** 使用**位掩码编码**法器类型。每种法器对应一个 bit 位，组合触发条件用位掩码表示，运行时通过位运算 `AND` 检测是否满足触发条件。7 种共鸣效果可叠加：

| 共鸣名称 | 触发条件 | 效果 |
|---------|---------|------|
| 烈火焚心 | 涡轮木鱼 + 电子烧香 | 点击×1.5，自动+2/秒 |
| 珠联璧合 | 量子佛珠 + 如意 | 暴击率+5%，暴击倍数×1.5 |
| 洞察先机 | 全视之眼 + 罗盘 | 效率+15% |
| 初心 | 基础木鱼 + 基础烧香 | 点击×2.0 |
| 佛光普照 | AI诵经机 + 袈裟 | 自动+5/秒，效率+20% |
| 三清合一 | 拂尘 + 如意 + 罗盘 | 点击+2/击，自动+2/秒，暴击率+8% |
| 万法归宗 | 全视之眼 + 量子佛珠 + AI诵经机 | 点击×3.0，暴击率+15%，暴击倍数×2.0 |

**`CreditRating`** 实现 **6 级信用体系**（S/A/B/C/D/F）。行为影响信用分：按时还款 +20、每日活跃 +2、逾期 -50、爆仓 -100。信用等级实时影响借贷利率、可借额度、最大杠杆与交易手续费。

### 2.5 策略与事件层

**`MarketEvent`** 包含 29 个校园主题事件，临时修改资产漂移率与波动率。业障值影响市场波动率：

```
volatility = base × (1 + yezhang / 2000)
```

业障 ≥ 1000 时概率触发"天劫"：drift = -15，volatility = 5.0，持续 20 秒，期间所有资产价格腰斩。

**`PortfolioAdvisor`** 基于**夏普比率**计算各资产的风险-收益比，在 `ExchangeDialog` 中绘制 **CAL 风险-收益图**：蓝色散点表示各资产（标准差×10⁴, 期望收益×10⁴），红色虚线射线从无风险利率点穿过最大夏普比率资产。

**`ArbitrageScanner`** 实时扫描因果私缘市场价格与理论价（0.6×众生法缘 + 0.4×善行福缘）的偏差，偏离超过阈值时提示套利机会。

**`AutoInvestor`** 按设定间隔和金额自动执行买入操作，支持定投开关控制。

### 2.6 教程系统层

**`TutorialManager`** 为单例状态机，管理 38 步教程流程。每步配置包含：文本、目标控件名、等待类型（`next`/`click_area`/`click_cloud`）、是否为对话框模式。

**`TutorialOverlay`** 为全屏遮罩 `QWidget`，使用 `QPainter::CompositionMode_Clear` 实现**高亮镂空效果**：全屏半透明暗色背景上，在目标控件位置"清除"遮罩，露出金色高亮边框。

```cpp
// 高亮镂空核心逻辑
painter.setCompositionMode(QPainter::CompositionMode_Clear);
painter.fillRect(targetRect, Qt::transparent);  // 在遮罩上"挖洞"
painter.setCompositionMode(QPainter::CompositionMode_SourceOver);
painter.drawRect(targetRect);  // 绘制金色边框
```

`findTutorialTarget()` 遍历控件树，过滤可见且尺寸 >50×50 的控件，避免命中 `.ui` 中的默认占位控件。气泡定位采用智能策略：目标在上半屏则气泡放下，下半屏则放上；对话框模式强制左下角定位防溢出。

## 四、项目总结与反思

### 4.1 项目亮点

1. **金融模拟与游戏化的深度融合**：将夏普比率、资本市场线、套利检测等专业金融概念以可视化方式呈现，降低了理解门槛。
2. **多层成长系统**：修炼等级、法器共鸣、信用等级、成就系统四条成长线并行，玩家在不同阶段都有明确的进阶目标。
3. **38 步内嵌教程系统**：相比传统的弹窗式教程，遮罩式实时引导让玩家在真实界面中边操作边学习，显著降低了上手门槛。
4. **古风仙侠的视觉包装**：统一配色（深褐+金色）、字体（行楷/楷体）、按钮规范，将金融界面与仙侠世界观有机结合。

### 4.2 遇到的困难与解决方案

| 困难 | 解决方案 |
|------|---------|
| 教程系统气泡截断 | 三重方案：强制 `layout()->activate()` + `adjustSize()`、字体降级（14px→12px）、阴影 padding 补偿 |
| 高亮目标控件偏移 | `findTutorialTarget()` 增加最小尺寸过滤（>50×50）和可见性检查 |
| 建筑热区误触 | 通过 `door_glow` 中心坐标精确计算门位置热区，将热区缩小至门区域 |
| 源代码归档后 `.pro` 路径失效 | 批量替换所有路径前缀为 `src/`，编译验证通过 |

### 4.3 待改进方向

1. **卖空机制**：当前仅支持买入做多，未来计划加入卖空与保证金风控系统。
2. **图形风格统一**：部分界面仍存在控件位置微调不足的问题，需进一步统一配色与字体规范。
3. **交互方式丰富**：当前以点击为主，未来可增加拖拽调整持仓、手势切换界面、快捷键操作等。
4. **多轮回历史**：当前仅保存单轮数据，未来计划加入多轮历史记录与排行榜。

### 4.4 课程收获

通过本项目，团队深入实践了 C++ 面向对象编程的核心思想：**继承与多态**（`Asset` 基类与四个派生类）、**单例模式**（`TutorialManager`）、**观察者模式**（Qt 信号槽）。同时积累了 Qt 界面开发经验，包括自定义绘制（`QPainter`）、布局管理、事件处理、定时器机制等。最重要的是，我们理解了"从需求到实现"的完整开发流程：**需求分析 → 原型设计 → 编码实现 → 测试迭代 → 文档归档**。
